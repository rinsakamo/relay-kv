# devlog_2026-04-14

## 今日やったこと

今日は、RelayKV の decode-time assisted path をさらに一段進めて、layer 14 を対象にした actual injection 実装の直前段階まで整理した。

出発点としては、前日までに

- gate
- hook
- tensor writeback readiness
- writeback tensor metadata
- injection target ready

まで到達していた。
今日のテーマは、その先にある actual injection の入口を、コード上ではっきり定義することだった。

まず、`relaykv_assisted_decode_layer14_prose_actual_runtime_tensor_selected.json` を確認し、状態遷移の整合性を見た。
ここでは、gate-pass step で

- `replacement_injection_selected = true`
- `replacement_injection_target = "layer14_attention_output"`
- `replacement_injection_mode = "actual_runtime_tensor_selected"`
- `replacement_injection_applied = true`

まで進んでいた一方で、初回は writeback 側の値が gate-fail 扱いで上書きされるズレがあり、そこを修正した。
修正後の再実行では、gate-pass step で

- `replacement_writeback_applied = true`
- `replacement_writeback_mode = "tensor_writeback_ready_only"`

がきれいに揃い、gate / hook / writeback / injection candidate の状態遷移が矛盾なく閉じることを確認した。

その次に、actual injection point の最小設計に進んだ。
ただし、いきなり Hugging Face の forward path を直接書き換えるのではなく、まずは

- actual injection を試みる helper を作る
- decode loop のどこでそれを呼ぶかを固定する
- 今はまだ wired していないことを明示する

という方針を取った。

このために、`apply_layer14_actual_injection(...)` という helper を追加した。
この関数はまだ本当の forward-path wiring は行わず、

- injection target が `layer14_attention_output` であること
- ここが将来の actual wiring callsite であること
- 現時点では `not_wired_yet` を返すこと

だけを明示する stub とした。

その上で、gate-pass かつ injection-ready の step では、decode loop 内でこの helper を実際に呼ぶようにした。
この変更により、step log には

- `replacement_injection_ready`
- `replacement_injection_ready_reason`
- `replacement_injection_apply_reason`
- `replacement_injection_applied`

が追加され、gate-pass step では

- `replacement_injection_ready = true`
- `replacement_injection_apply_reason = "not_wired_yet"`
- `replacement_injection_applied = false`
- `replacement_injection_mode = "actual_injection_callsite_defined"`

となるように整理した。

一方、gate-fail step では

- `replacement_injection_ready = false`
- `replacement_injection_ready_reason = "gate_not_passed"`
- `replacement_injection_apply_reason = "gate_not_passed"`
- `replacement_injection_applied = false`

となるように揃えた。
これにより、gate fail と not-yet-wired の違いがログ上ではっきり区別できるようになった。

最終的に `relaykv_assisted_decode_layer14_prose_actual_injection_callsite_defined.json` では、次のことが確認できた。

- step 0 / 9 は gate fail のため full-path fallback
- gate-pass step では injection target は `layer14_attention_output`
- gate-pass step では injection-ready は true
- ただし actual wiring はまだ未実装のため `replacement_injection_apply_reason = "not_wired_yet"`
- `replacement_injection_applied = false`
- run-level summary では `injection_stage = "actual_injection_callsite_defined"`

つまり、actual wiring の入口そのものはコード上に定義されたが、まだ本当の forward-path 差し替えはしていない、という段階に到達した。

## 今日の整理

今日の終わり時点で、layer 14 の assisted decode path は次の段階まで到達した。

1. actual runtime tensor selected
2. actual injection attempted but not wired
3. actual injection callsite defined

ここまで来たことで、次に残っている本質的な作業はかなり限定された。

## 現時点の理解

今日の時点での理解は次の通り。

- layer 14 は引き続き first assisted replacement 候補として最も安全
- gate 条件は decode-time でも機能しており、危ない step を除外できている
- writeback tensor と injection target は step ごとに安定して確定できる
- actual injection の callsite はコード上で定義できた
- したがって残る中心課題は、`apply_layer14_actual_injection(...)` の中身を実 forward path にどう接続するか、にほぼ絞られた

## 次にやること

次の実装テーマは、`apply_layer14_actual_injection(...)` の中身を本当に wiring すること。

ただし、最初はかなり狭く進めるのがよい。

- layer 14 only
- gate-pass かつ injection-ready step only
- failure 時は full-path fallback
- 1 箇所の hook point にだけ注入する

次回は、

- Hugging Face モデルのどこに hook を刺すか
- `layer14_attention_output` 相当のどのテンソルを差し替えるか
- wiring failure をどうログするか

を決めて、actual forward-path wiring に入る。
