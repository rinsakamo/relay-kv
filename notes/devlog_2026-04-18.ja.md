## 今日やったこと

layer14 assisted decode prototype を、runtime bridge ready から next-step apply prototype まで進めた。

### 1. runtime bridge ready の整備
- KV-head 空間の approx attention output `[1,2,1,128]` を hidden-state 空間 `[1,1,1536]` に bridge
- bridge hidden state を `o_proj` pre-hook target の shape / device / dtype に合わせる runtime bridge helper を追加
- injection target を `layer14_attention_output` から `layer14_oproj_input` に更新
- injection readiness を 3D hidden-state 契約に更新

### 2. apply callsite 契約の整合
- `prepare_layer14_actual_injection(...)` と `apply_layer14_actual_injection(...)` の target 名を `layer14_oproj_input` に揃えた
- これにより `ready=true` かつ `apply_reason="not_wired_yet"` の状態まで確認できた

### 3. same-step apply 失敗原因の切り分け
- `o_proj` pre-hook は `model(...)` 実行中に発火する
- 一方、approx tensor の計算と armed は forward 後にしかできない
- そのため same-step apply は時系列的に成立しないことを確認

### 4. next-step apply prototype の実装
- injection state を step をまたいで保持する方式へ変更
- gate-pass + injection_ready の step で `armed_for_next_step`
- 次 step の `o_proj` pre-hook で pending injection を consume し、1回だけ差し替えて自動 disarm
- `replacement_oproj_pre_hook_replaced` を追加して、実際に pre-hook 置換が起きたかを記録

## 実行結果

軽い確認実行では以下を確認できた。

- step 2: `ready=true`, `replaced=true`
- step 3: `ready=true`, `replaced=true`

これにより、next-step apply prototype が成立していることを確認した。
