# devlog_2026-04-13

## 今日やったこと

今日は、RelayKV の decode-time 実装をさらに一段進めて、layer 14 を対象にした assisted decode の設計と状態遷移をかなり細かく固めた。

まず、actual assisted decode に入る前の実装差分を整理した。
いきなり model 全体を書き換えるのではなく、

- shadow
- shadow+compare
- assisted-ready
- gated decision
- hooked dry-run
- logical writeback
- tensor writeback readiness
- injection-target ready

という段階に分けて進める方針を明文化した。
特に、layer 14 を最初の actual replacement 候補として扱い、layer 27 は後回しにする方針を維持した。

次に、`notes/decode_time_relaykv_design.md` に `Layer-14 Gated Actual Writeback Plan` を追加した。
この節では、

- writeback target は approximate layer-14 attention output
- first version は layer 14 only
- gate-pass step のみ
- full-path fallback を常に持つ
- decision / hook / writeback をログ上で分離する

という設計を明確にした。

その後、`scripts/run_relaykv_assisted_decode_layer14_hooked.py` を中心に、layer 14 の gated path をさらに細かく整理した。

まず、`replacement_applied` という名前が紛らわしかったので、意味をはっきりさせるために `replacement_selected` に変更した。
これで状態の意味が

- replacement_selected
- replacement_hook_triggered
- replacement_writeback_applied

の3段に整理された。

次に、logical writeback の段階に進めた。
ここでは、gate-pass step に対して

- replacement_selected = true
- replacement_hook_triggered = true
- replacement_writeback_applied = true
- replacement_writeback_mode = `logical_writeback_only`

としつつ、まだ actual tensor writeback はしていない状態を明示した。
その後、mode 名をさらに整理し、`logical_writeback_only` を `tensor_writeback_ready_only` に揃えた。

さらに、actual tensor writeback に入る前段として、approx attention output を runtime tensor として保持し、shape / device / dtype の readiness check を通す段階まで進めた。
ここでは

- `_full_attention_output`
- `_approx_attention_output`

を内部用に保持し、
`check_tensor_writeback_ready(...)` を追加して、

- shape mismatch
- device mismatch
- dtype mismatch

がないか確認するようにした。

その結果、layer 14 の gate-pass step では `replacement_tensor_writeback_ready = true` が得られ、gate-fail step ではきれいに `false` になることを確認した。
この時点で、actual writeback に必要な tensor の準備が runtime 上で整った。

次に、writeback tensor 自体の metadata を step log に残すようにした。
具体的には、gate-pass step で

- `replacement_writeback_tensor_shape`
- `replacement_writeback_tensor_device`
- `replacement_writeback_tensor_dtype`

を記録した。
再実行後の出力では、gate-pass step で一貫して

- shape = `[1, 2, 1, 128]`
- device = `cpu`
- dtype = `torch.float32`

が出ており、writeback candidate tensor の性質が step ごとに確認できる状態になった。

その後、actual injection point の最小実装に入る前段として、injection target を step ごとに確定する段階に進めた。
ここでは、gate-pass かつ tensor-writeback-ready の step に対して

- `replacement_injection_selected = true`
- `replacement_injection_target = "layer14_attention_output"`
- `replacement_injection_mode = "planned_runtime_injection_only"`

を記録するようにした。
一方、gate-fail step では

- `replacement_injection_selected = false`
- `replacement_injection_target = null`
- `replacement_injection_mode = "gate_not_passed"`

になるよう整理した。

最終的に、`relaykv_assisted_decode_layer14_prose_injection_target_ready.json` では、

- gate-fail step 0 / 9 は full path fallback
- gate-pass step では writeback tensor metadata が入る
- gate-pass step では injection target が `layer14_attention_output` に固定される
- run-level summary にも `injection_stage = "planned_runtime_injection_only"` が残る

ところまで確認できた。

途中で `SyntaxError` や `step_logs` の扱いのズレがあり、一度だけ最後の step しか保存されない状態になったが、これは

- 二重 `else`
- summary dict のカンマ抜け
- append 位置の確認

などを直して解消した。
また、`evaluate_layer14_step(...)` の return でキーが重複していた箇所も整理した。

## 今日の整理

今日の終わり時点で、layer 14 の decode-time assisted path は次の段階まで到達した。

1. gated replacement decision
2. hooked dry-run
3. logical writeback
4. tensor writeback readiness
5. writeback tensor metadata
6. injection-target ready

つまり、残っている本質的な作業はかなり絞られている。

## 現時点の理解

今日の時点での理解は次の通り。

- layer 14 は引き続き first assisted replacement 候補として妥当
- gate 条件は機能しており、危ない step を除外できている
- gate-pass step では actual writeback に必要な tensor の shape / device / dtype が揃っている
- writeback candidate tensor は step ごとに runtime object として確定できている
- injection target は `layer14_attention_output` に固定できた
- したがって、残る課題は「その tensor を decode path のどこに注入するか」にほぼ限定された
