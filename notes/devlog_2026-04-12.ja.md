# devlog_2026-04-12

## 今日やったこと

今日は、RelayKV の decode-time 系プロトタイプをかなり前に進めた。

まず、quick benchmark 系を整理した。`repetitive / prose / structured` の 3 prompt style を揃え、cross-style の比較表を作成した。現時点の inspected setup では、`very-heavy (1,1,7)` が 3 条件で一貫して `uniform` より良かった。特に hard deep layer である layer 27 の改善が大きく、layer-wise budget allocation が軽量な scoring 微調整より強いレバーであることを再確認した。

そのうえで、decode-time 系の実装準備に進んだ。最初に `notes/decode_time_relaykv_design.md` を整理し、decode-time の実装方針を明文化した。ここでは以下をはっきり決めた。

- first decode-time prototype は representative layers (`0, 14, 27`) から始める
- fixed-prefill cold blocks を採用する
- decode で追加された token は hot-only として扱う
- first assisted-decode 前に shadow retrieval と step-wise compare を挟む

次に、`scripts/run_relaykv_decode_prototype.py` を使って shadow-mode の decode-time prototype を作った。この版では manual greedy decode loop を回しつつ、各 step で representative layers の retrieval を走らせ、selected blocks / candidate size / working size を記録した。`prose` と `structured` の 2 条件で step-level summary まで作成し、decode-time retrieval behavior を観察できる状態にした。

この shadow prototype から、いくつか重要な観察が得られた。

- layer 0 はかなり stable で recent-heavy
- layer 14 も mostly recent-heavy だが、boundary 付近で少し揺れる
- layer 27 は decode-time でも最も動的で、non-contiguous retrieval が多い
- `structured` は `prose` より recent-local / contiguous 寄り
- `prose` は layer 27 で飛び地 retrieval がより多い

さらに、`run_relaykv_decode_prototype.py` を shadow+compare に拡張し、decode step ごとの attention approximation error を representative layers で記録できるようにした。これで、retrieval summary だけでなく step-wise error まで見えるようになった。

この compare 結果がかなり重要だった。layer ごとの差が明確で、

- layer 14 は most steps で誤差がかなり小さい
- layer 0 は多くの step で小さいが occasional spike がある
- layer 27 は hardest で、step によってかなり大きく揺れる

ことが分かった。この結果から、first actual assisted decode は layer 14 から始めるのが最も安全だと判断した。

その後、`scripts/run_relaykv_assisted_decode_layer14.py` を作成し、layer 14 only の assisted-ready prototype を実装した。この版では、各 step で layer 14 の working KV と full/approx attention compare を記録し、replacement-ready な実験経路を確保した。結果として、layer 14 はやはりかなり安定していて、actual replacement 候補として妥当であることを確認した。

次に、layer 14 only の gated replacement 判定を実装した。gating 条件はかなり単純にして、

- contiguous candidate
- single selected block
- selected block id in `{14, 15}`
- `mean_abs_diff <= 0.002`

とした。この条件では 16 step 中 14 step が通過し、危なそうだった step 0 と step 9 だけが落ちた。想定どおりの挙動だった。

最後に、ファイル名を `scripts/run_relaykv_assisted_decode_layer14_hooked.py` として、hooked dry-run まで進めた。この版では、gate-pass step に対して runtime 上で replacement hook を trigger できるようにしつつ、まだ writeback は行わない `dry_run_no_writeback` とした。これにより、

- gated decision
- runtime hook
- 将来の actual writeback point

が切り分けられた状態になった。

## 現時点の理解

現状の RelayKV prototype について、今日の時点での理解は以下。

- quick benchmark の best tested config は、引き続き `very-heavy (1,1,7)`
- decode-time retrieval behavior は layer-dependent
- layer 27 は hardest で、decode-time でも non-contiguous retrieval が多い
- layer 14 は very simple / recent-local で、first assisted replacement の候補として強い
- actual assisted decode は、全 layer 一律ではなく段階的に導入すべき
- writeback 前に decision と hook を分離したのは正解だった
