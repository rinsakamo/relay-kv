# RelayKV ローカル中心運用ルール

## 基本方針
- 本線はローカルで進める
- 実験の本実行は通常シェルで行う
- `work` を実験ブランチ、`main` を安定ブランチとして使う
- Cloud Codex は原則使わない
- Codex CLI は必要時のみ使う

## 役割分担
### 通常シェル
- `git checkout`
- `.venv` 有効化
- 実験コマンド実行
- `jq` による結果確認
- commit

### Codex CLI
- repo 読解
- 最小差分パッチ提案または小修正
- 実行コマンド整理
- ログ解釈補助

### Cloud Codex
- 原則使わない
- 使う場合でも本線にはせず、参考差分扱いにする

## Git 運用
- 通常は `main` と `work` の2本だけ
- 一時ブランチは必要なときだけ作る
- `codex/...` ブランチは本線にしない
- Cloud Codex の PR は参考差分として扱う

## 変更の進め方
1. `work` で最小差分を入れる
2. ローカルで実験を回す
3. `jq` で確認する
4. 問題なければ commit
5. 区切りが良ければ `main` に取り込む

## Codex CLI の使い方
### 原則
- 読み取り優先
- 実験を自動実行させない
- 複数ファイルの大改造をさせない
- 1回1自由度で進める

### 安全な依頼例
- 現在の実行フローを repo ベースで説明して
- このエラーの最有力原因を最小差分で直して
- この commit の変更点を整理して
- 次に打つローカル実行コマンドと `jq` を出して

## AGENTS.md の方針
- 削除しなくてよい
- ただし Cloud 前提ではなく、ローカル中心に書き換える
- `AGENTS.md` は Codex CLI 向けのローカル運用ルールとして残す

## AGENTS.md に入れるべき内容
- scripts/ を先に見て、次に relaykv/ を見る
- repo 上の事実を優先する
- 最小差分を優先する
- 実験本実行はローカルで行う
- Codex は原則、実験を自動実行しない
- 出力順を固定する

## 出力順
1. 見たファイル / 見るべきファイル
2. 現在の理解
3. 問題点 or 実装ポイント
4. 最有力の原因候補
5. 最小修正案
6. 実行コマンド / jq / 確認ポイント
