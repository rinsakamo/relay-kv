# devlog_2026-04-16.ja

## 今日やったこと

今日は、RelayKV の decode-time assisted path を actual injection に近づけるために、
KV-head 表現から hidden-state 表現へ持ち上げる bridge の最小実装まで進めた。

出発点では、前日までに次のことが分かっていた。

- `compare_attention_outputs(...)` は `softmax(QK^T)V` の head-wise 出力をそのまま返している
- 実際の `writeback_tensor` は `[1, 2, 1, 128]`
- Qwen/Qwen2.5-1.5B-Instruct では `num_key_value_heads = 2`, `head_dim = 128`, `hidden_size = 1536`
- したがって、今の `writeback_tensor` は KV-head 空間の表現であり、HF 側 hook で見える hidden-state 1536 次元表現とは別空間である

この理解を受けて、今日はまず actual wiring の前提として
「KV-head 空間 → hidden-state 空間」の bridge を作る方針に切り替えた。

最初に、`compare_attention_outputs(...)` の中身を確認した。
そこでは `scaled_dot_product_attention(...)` が

- query: `[1, heads, 1, head_dim]`
- key: `[1, heads, seq_len, head_dim]`
- value: `[1, heads, seq_len, head_dim]`

を受けて、
そのまま `[1, heads, 1, head_dim]` の出力を返していた。
flatten や output projection は入っていなかった。
このため、今見ている `full_output` / `approx_output` は projection 前の head-wise attention 出力であることがはっきりした。

次に、前回までに取っていた probe も踏まえて整理し直した。
HF 側では

- `self_attn` 出力
- `layer` 出力
- `o_proj` 前
- `o_proj` 後

のすべてが `[1, 1, 1536]` で観測されており、
hook point を増やしても 256 次元表現は見つからなかった。
そのため、hook point 探索をこれ以上広げるより、bridge を作る方が正しいと判断した。

そのうえで、`kv_heads_to_hidden_state_bridge(...)` を新たに入れた。
この helper では、

- 入力 `[B, kv_heads, Q, head_dim]`
- `num_attention_heads / num_key_value_heads` の比率で KV heads を repeat
- `[B, attn_heads, Q, head_dim]`
- それを `[B, Q, attn_heads, head_dim]` に並べ替え
- 最後に flatten して `[B, Q, hidden_size]`

へ変換するようにした。

Qwen2.5-1.5B-Instruct では

- attention heads = 12
- KV heads = 2
- repeat factor = 6

なので、
`[1, 2, 1, 128]` を `[1, 12, 1, 128]` に展開し、
最終的に `[1, 1, 1536]` に持ち上げる最小 bridge になっている。

さらに `run_decode_loop_layer14_assisted_ready(...)` に

- `num_attention_heads`
- `num_key_value_heads`

を渡すようにして、
gate-pass かつ tensor-writeback-ready の step では
`writeback_tensor` から bridge hidden state を作る処理を入れた。

その結果、gate-pass step では一貫して

- `replacement_writeback_tensor_shape = [1, 2, 1, 128]`
- `replacement_bridge_hidden_state_shape = [1, 1, 1536]`
- `replacement_bridge_hidden_state_device = "cpu"`
- `replacement_bridge_hidden_state_dtype = "torch.float32"`

となり、
`replacement_bridge_compare` にも

- writeback tensor shape
- bridge hidden-state shape
- expected hidden size = 1536

を残せるようになった。

run-level summary も更新し、
`decode_policy.injection_stage` と `replacement_injection.mode` を
`kv_head_bridge_ready` まで進めた。
これで、まだ実際の runtime 注入はしていないが、
少なくとも shape 上は hidden-state 表現の注入候補を作れる段階に進んだ。

今回の時点では、bridge tensor はまだ

- device = cpu
- dtype = torch.float32

であり、
HF 側 hook で観測される

- device = cuda:0
- dtype = torch.float16

には合わせていない。
したがって今日の達成点は

- shape bridge は完成
- runtime bridge は未実装
- actual replacement wiring も未実装

という段階である。

## 今日の整理

今日の終わり時点で、layer 14 の decode-time assisted path は次の段階まで到達した。

1. compare 出力が projection 前の head-wise attention 出力であることを確認
2. hook point 探索では hidden-state 1536 空間しか見えないことを再確認
3. KV-head 表現 `[1, 2, 1, 128]` を hidden-state 形 `[1, 1, 1536]` に持ち上げる helper を追加
4. gate-pass step で bridge hidden state を作れるようにした
5. run-level stage を `kv_head_bridge_ready` に進めた

## 現時点の理解

今日の時点での理解は次の通り。

- `writeback_tensor` は GQA の KV-head 空間にある head-wise attention 出力である
- HF 側 forward path で見える実テンソルは hidden-state 1536 次元空間にある
- よって actual injection の前には空間の橋渡しが必要
- 最小の repeat-based bridge で `[1, 1, 1536]` を作ること自体はできた
- ただしこの bridge はまだ runtime の device/dtype に合わせていない
- したがって次の課題は、bridge hidden state を `cuda:0 / float16` に合わせて runtime 注入候補にすること
