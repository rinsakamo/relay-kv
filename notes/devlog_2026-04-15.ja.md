# devlog_2026-04-15

## 今日やったこと

今日は、RelayKV の decode-time assisted path について、actual wiring に入る前の「表現の対応関係」をかなりはっきりさせた。

出発点では、layer 14 の gated assisted path について

- gate
- hook
- tensor writeback readiness
- injection target ready
- actual injection callsite defined

まで進んでいたが、まだ `writeback_tensor` を Hugging Face 側の実 forward path にどう接続するかが不明だった。
今日の中心テーマは、その接続先の探索と、`writeback_tensor` 自体の表現の正体確認だった。

まず、`scripts/run_relaykv_assisted_decode_layer14_hooked.py` の整備を続けた。
hook state の持ち方を見直し、

- `self_attn` 用の probe
- layer 14 block 全体用の probe

を分けて記録できるようにした。
これにより、単一 hook ではなく dual hook probe によって、

- `replacement_attn_hook_*`
- `replacement_layer_hook_*`

を step ごとに残せるようにした。

その過程で、いくつかコード上の不整合も直した。

- hook state の field 名変更にともなう参照漏れ
- `if injection_ready:` 周辺の崩れたインデント
- hook 観測結果が gate-fail 側にしか残らない問題
- `layer14_probe_handle.remove()` が return の後にあって実行されない問題

などを整理した。
この結果、gate-pass / gate-fail の両方で hook 情報が一貫して記録されるようになった。

次に、dual hook probe の実行結果を確認した。
結果として、gate-pass step でも gate-fail step でも一貫して

- `replacement_attn_hook_shape = [1, 1, 1536]`
- `replacement_layer_hook_shape = [1, 1, 1536]`
- dtype は `torch.float16`
- device は `cuda:0`

であることが分かった。

この結果は重要だった。
少なくとも今回見ている範囲では、

- `self_attn` hook 出力
- layer 14 block hook 出力

のどちらも hidden-state 空間の 1536 次元表現に見えており、hook point を少し広げても、今の `writeback_tensor` と同じ表現にはならなかった。

その一方で、writeback 側は gate-pass step で一貫して

- `replacement_writeback_tensor_shape = [1, 2, 1, 128]`
- dtype は `torch.float32`
- device は `cpu`

だった。
shape compare / representation compare を入れたことで、この差は step ごとに明示的に見えるようになった。

ここでさらに前に進むために、`writeback_tensor` 自体の内訳をログに出すようにした。
`attention_tensor_breakdown` を追加して、

- full / approx の shape
- num_heads
- head_dim
- num_heads * head_dim

を記録した。
さらに run-level summary に `model_attention_layout` を追加し、

- hidden_size
- num_attention_heads
- num_key_value_heads
- head_dim
- kv_hidden_size

を残すようにした。

その結果、Qwen/Qwen2.5-1.5B-Instruct の layer 14 については

- hidden_size = 1536
- num_attention_heads = 12
- num_key_value_heads = 2
- head_dim = 128
- kv_hidden_size = 256

であることが確認できた。
そして `attention_tensor_breakdown` は full / approx ともに一貫して

- num_heads = 2
- head_dim = 128
- flattened_last_dim = 256

だった。

この結果から、今日の一番大きな収穫が得られた。

今の `compare_attention_outputs(...)` が返している `writeback_tensor` は、
HF 側 hook で観測される hidden-state 表現ではなく、

- `num_key_value_heads × head_dim = 2 × 128 = 256`

に対応する **KV-head 表現** である可能性がかなり高い。

一方で、HF 側の `self_attn` / layer hook で見えているのは

- hidden_size = 1536

の **hidden-state 表現** である。
この 2 つは同じ表現空間ではないので、今の `writeback_tensor` をそのまま `self_attn` や layer 出力に注入するのは正しくない、という結論に至った。

つまり、今日の時点での理解は、

- hook point をもう少し変えるだけでは解決しない
- 本当に必要なのは、KV-head 空間から hidden-state 空間への橋渡し
- 実 wiring の前に、`writeback_tensor` が attention 内部のどの表現に対応するのかをさらに詰める必要がある

というものになった。

## 今日の整理

今日の終わり時点で、layer 14 の decode-time assisted path については次のことが明確になった。

1. dual hook probe により、`self_attn` と layer block の両方の hook 出力を比較できるようになった
2. 両 hook 出力は一貫して hidden-state 1536 次元だった
3. `writeback_tensor` は一貫して `[1, 2, 1, 128]` の KV-head 形だった
4. model config と tensor breakdown を突き合わせることで、`writeback_tensor` が `num_key_value_heads × head_dim` に対応することが確認できた
5. したがって、次の課題は hook point の微調整ではなく、KV-head 表現を hidden-state 表現へ対応づける設計に移った

## 現時点の理解

今日の時点での理解は次の通り。

- layer 14 の gate 自体は引き続き安定して機能している
- gate-pass step では writeback-ready / injection-ready までは一貫して成立する
- しかし actual forward-path wiring の問題は、単なる hook location の問題ではない
- 今の `writeback_tensor` は hidden-state ではなく KV-head 表現である
- HF モデル側の実モジュール出力は hidden-state 1536 次元で観測される
- したがって、actual injection に進む前に、表現変換または内部表現の再定義が必要

## 次にやること

次の実装テーマは、KV-head 表現と hidden-state 表現の橋渡しをどう定義するかを詰めること。

現時点で自然な次の候補は、

- attention 内部で output projection 前後のどこを probe すべきかをさらに確認する
- `writeback_tensor` が attention 内部のどの計算段階に対応するのかを確認する
- 必要なら KV-head 表現から hidden-state 表現への変換 helper を設計する

の3つである。

いまの段階は、

- gate
- hook
- dry-run
- callsite defined
- dual hook probe
- tensor layout probe

まで進んでおり、次回は
**KV-head → hidden-state bridge design**
に入る。
